# Governance Reference

## 样式与资源作用域归属

业务资产不应随意污染全局共享空间，以此杜绝死代码（Dead Code）残留：

- **模块私有资源就近**：模块专属的配图、Excel 导入下载模板、局部定制样式，一律放入模块内部对应的 `assets/` 或 `styles/` 下。
- **全局公共资产上移**：只有全局通用的系统 Logo、通用的 Empty 空状态插画，才可以放入全局 `src/shared/assets/`。
- **全局主题治理**：全局主题变量、Reset 样式、Tailwind 配置统一放入 `shared/styles/` 或项目既有样式包中。

## 国际化 (i18n) 就近治理

大型系统中，防范几万行文案堆在一个大 JSON 中造成的维护与检索灾难：

- **就近存放**：大型业务推荐模块就近，在模块内建立 `locales/zh-CN.ts`、`locales/en-US.ts`。小项目文案可统一放在 `shared/locales/`。
- **命名空间隔离**：i18n 的 key 命名必须深度绑定业务域前缀，杜绝全局语义冲突：

```text
system.user.form.username
system.user.action.resetPassword
```

## 环境变量隔离封装

规范运行时配置、构建配置与业务常量。禁止在业务组件或业务局部逻辑中直接散落读取、硬编码底层环境变量（如直接读取 `import.meta.env`）。

合规方案：API baseURL、租户、加密等平台配置必须由 `app/config` 或 `services/config` 进行业务语义封装，业务层面向语义配置对象进行声明式调用。业务局部枚举和默认值放模块内 `constants.ts`。

## 测试与 Mock 跟随

- **单测就近**：单元测试贴近组件、hooks、utils 存放。
- **e2e 集中**：端到端测试按业务场景放全局 `tests/e2e/`。
- **Mock 隔离**：Mock 数据优先放模块内 `__mocks__/` 或 `mocks/`，只有发生跨模块 Mock 时才考虑上移。

## 业务模块自解释文档规范

凡是复杂的、包含子模块或多权限联动的业务模块，根目录下必须配备 `README.md`。该文档是团队开发者和 AI 快速理解模块边界的唯一"说明书"，必须包含：

1. **模块边界**：当前模块负责处理什么核心业务，不处理什么。
2. **页面入口**：主视图和各路由视图的映射关联。
3. **API 与核心接口**：模块对外的请求核心清单。
4. **权限码清单**：当前业务域下绑定的核心菜单与按钮权限码。
5. **共享依赖说明**：当前模块是否有对其他非底层模块的弱依赖关系。
6. **维护注意**：特殊业务流程或历史技术债规避说明。
